# `pytest-multi-results-action`

[![BSD-3-Clause License](https://img.shields.io/github/license/yeoldegrove/light-the-torch)](https://opensource.org/licenses/BSD-3-Clause)
[![Project Status: WIP](https://www.repostatus.org/badges/latest/wip.svg)](https://www.repostatus.org/#wip)

> **Note**: This is a fork of the excellent [pytest-results-action](https://github.com/pmeier/pytest-results-action) by [@pmeier](https://github.com/pmeier), enhanced with multi-suite aggregation capabilities for complex CI/CD pipelines.

`pytest-multi-results-action` is a Javascript action for [GitHub Actions](https://github.com/features/actions) to surface failing [`pytest`](https://docs.pytest.org) tests without crawling through logs. It supports multi-suite aggregation, making it perfect for complex CI/CD pipelines with multiple test environments.

## Why do I need it?

For all the awesomeness GitHub Actions brings, compared to other CI providers, logs above a few thousands lines load painfully slow. This can happen quite fast in larger test suites and is almost a given if one runs `pytest` in verbose mode, i.e. with `-v` set.

In addition to loading faster, other CI providers have an option to surface failing tests so there is no need to look at the logs in most cases. For example, on CircleCI using the builtin step [store_test_results](https://circleci.com/docs/collect-test-data/) is sufficient.

`pytest-multi-results-action` aims to bring the same UX to GitHub Actions, with enhanced support for:

- **Multi-suite aggregation**: Combine results from multiple test environments (e.g., chroot, QEMU, cloud, OCI)
- **Consolidated reporting**: Get an overview of all test suites with individual suite details
- **Flexible input patterns**: Support for wildcards and multiple file patterns

## How do I use it?

To be able to surface failing tests, `pytest-multi-results-action` parses JUnit XML files generated by `pytest`. For this the `--junit-xml` option needs to be set. The same value(s) need to be passed to the action inputs.

### Multi-Suite Aggregation

For workflows with multiple test environments, use the `files` input:

```yaml
- name: Download all test artifacts
  uses: actions/download-artifact@v4
  with:
    pattern: "*-test-ng-*"
    path: test-artifacts
    merge-multiple: true

- name: Generate Consolidated Test Report
  if: always()
  uses: yeoldegrove/pytest-multi-results-action@main
  with:
    # Multiple JUnit XML file patterns
    files: |
      test-artifacts/**/*.test-ng.xml
      test-artifacts/**/*.test.xml
    title: "Test Results"
    summary: true
    display-options: "fEX"
    fail-on-empty: false
```

## Input Parameters

| Input                    | Description                                                                        | Required | Default                           |
| ------------------------ | ---------------------------------------------------------------------------------- | -------- | --------------------------------- |
| `files`                  | Multiple JUnit XML file patterns for multi-suite aggregation                       | Yes      | -                                 |
| `summary`                | Add a summary of the results at the top of the report                              | No       | `true`                            |
| `display-options`        | Select which results should be included in the report (follows `pytest -r` syntax) | No       | `fEX`                             |
| `fail-on-empty`          | Fail the workflow if no JUnit XML was found                                        | No       | `true`                            |
| `title`                  | Title of the test results section in the workflow summary                          | No       | `Test results`                    |
| `metadata-fields`        | Comma-separated list of metadata fields to extract from XML and display in tables  | No       | `""` (defaults to "Suite" column) |
| `metadata-field-mapping` | JSON object mapping XML attribute names to display names                           | No       | `{}` (empty object)               |

### Test Metadata Support

The action supports extracting and displaying custom metadata from JUnit XML files. This is particularly useful for multi-environment testing where you want to track additional information about each test suite.

#### Using pytest-metadata

The easiest way to add metadata is using the [pytest-metadata](https://pypi.org/project/pytest-metadata/) plugin, which automatically includes metadata in JUnit XML output:

```bash
# Install pytest-metadata
pip install pytest-metadata

# Run tests with metadata
pytest --metadata Artifact=aws-gardener_prod-amd64-2010.0-6f7a7b19 \
        --metadata Type=chroot \
        --metadata Namespace=test-ng \
        --junit-xml=results.xml
```

The metadata will be automatically included in the JUnit XML as `<property>` tags:

```xml
<testsuite name="pytest" ...>
  <properties>
    <property name="Artifact" value="aws-gardener_prod-amd64-2010.0-6f7a7b19"/>
    <property name="Type" value="chroot"/>
    <property name="Namespace" value="test-ng"/>
  </properties>
  <!-- test cases -->
</testsuite>
```

#### Using Metadata in GitHub Actions

```yaml
- name: Generate Test Report with Metadata
  if: always()
  uses: yeoldegrove/pytest-multi-results-action@main
  with:
    files: |
      test-artifacts/**/*.qemu.test-ng.xml
      test-artifacts/**/*.chroot.test-ng.xml
      test-artifacts/**/*.oci.test-ng.xml
    title: "Test Results"
    summary: true
    display-options: "fEX"
    fail-on-empty: false
    # Specify which metadata fields to display (in order)
    metadata-fields: "Artifact,Type,Namespace"
    # Optional: customize display names
    metadata-field-mapping: '{"Artifact": "Test Artifact", "Type": "Test Type", "Namespace": "Test Namespace"}'
```

#### Metadata Features

- **pytest-metadata Integration**: Automatically extracts metadata from pytest-metadata properties
- **Automatic Extraction**: Metadata is automatically extracted from XML attributes and properties
- **Fallback to Filename**: If metadata is not present in XML, the action falls back to extracting information from the filename
- **Custom Field Mapping**: Map XML attribute/property names to custom display names
- **Flexible Display**: Choose which metadata fields to display in the table and their order
- **Multi-Suite Support**: Metadata is displayed per test suite in the summary table
- **Suite Fallback**: If no metadata fields are specified, displays a "Suite" column with the suite name
- **Order Preservation**: Metadata fields are displayed in the order specified in the `metadata-fields` input

### Display Options

The `display-options` parameter follows the same syntax as `pytest -r`:

- `f` - Show failed tests
- `E` - Show error tests
- `s` - Show skipped tests
- `x` - Show xfailed tests
- `X` - Show xpassed tests
- `p` - Show passed tests
- `P` - Show passed tests (same as `p`)
- `a` - Show all tests
- `N` - Reset the list (only chars after last `N` are used)

## Report Output

The report will be posted to the workflow summary. For multi-suite aggregation, the report includes:

1. **Overall Summary**: Aggregated statistics across all test suites
2. **Suite Table**: Overview of each test suite with individual statistics
3. **Detailed Results**: Per-suite breakdown of failed, errored, and xpassed tests

### Multi-Suite Report Structure

When using the `files` input with multiple test suites, the report structure includes:

- **Overall Summary**: Total tests, duration, and aggregated results across all suites
- **Test Suite Table**: Individual suite statistics (total, passed, failed, skipped, errors, duration)
- **Suite Details**: Detailed breakdown for each suite with relevant test failures/errors

This provides both high-level overview and detailed insights into specific test environments.

## Development and Testing

### Getting Started

1. **Run tests:**

   ```bash
   # Basic test with default settings
   npm run test
   ```

# Test multiple files

npm run test

# Test multi-file aggregation

npm run test-multi

# Test specific environments

npm run test-chroot
npm run test-oci

````

2. **Build the action:**

```bash
npm run build
````

3. **Format code:**
   ```bash
   npm run format
   ```

### Test Output

All test runs generate output in the `test-output/` directory:

- `test-output/report.md` - Generated markdown report
- Console output shows the report content and any debug information

### Available Test Commands

| Command                 | Description                      |
| ----------------------- | -------------------------------- |
| `npm run test`          | Basic test with default settings |
| `npm run test-metadata` | Test with metadata passed        |
